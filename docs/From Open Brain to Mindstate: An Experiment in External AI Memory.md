# From Open Brain to Mindstate: An Experiment in External AI Memory

Nathan Jones recently published a thoughtful video discussing what he calls the “Open Brain” (See: [https://youtu.be/2JiMmye2ezg](https://youtu.be/2JiMmye2ezg) ). The argument is simple but important. As \#AI tools become part of daily work, the context they rely on remains scattered across separate systems. Each model or product keeps its own fragment of memory, which forces users to constantly rebuild understanding from scratch. Jones proposes a different direction: **move memory outside the model and into a persistent, AI-accessible store that can be shared across tools through an open interface.**

Nathan Jones is right about the central issue. If AI is going to be used seriously over time, memory cannot remain trapped inside separate products. It cannot sit partly in ChatGPT memory, partly in Claude projects, partly in a coding agent, and partly in scattered notes. That arrangement creates repeated prompting, lost context, and unnecessary friction. His “Open Brain” argument gets the strategic point right: memory should be external to the model, durable, queryable, and accessible across tools.

That idea matters, even if the technical ingredients are not new. A persistent store, embeddings, semantic retrieval, metadata extraction, and an access layer through MCP or a similar protocol are all familiar patterns. The value of Jones’ position is not novelty at the level of primitives. It is that he places those primitives in the right part of the system. Memory should belong to the user or the team, not to the application that happened to capture the latest interaction.

Still, the Open Brain framing is weaker at the implementation level than at the strategic level. A vector-backed note store exposed through MCP is useful, but it is not yet a robust memory system. It gives recall, but not necessarily coherence. It helps find similar fragments, but not always current state, decision history, contradiction, or structural relations. It also risks becoming a dumping ground if nothing enforces provenance, consolidation, and separation between source material and derived summaries.

That is where a more explicit system design becomes necessary. The key move is to stop treating memory as “RAG plus protocol” and start treating it as an information system with several layers. There is a canonical layer for captured material and provenance. There is a semantic layer for similarity-based retrieval. There is a structural layer for entities, links, projects, people, and decisions. Then there is an interface layer that exposes memory as usable behavior rather than raw storage. Agents should not need to think in terms of tables, vectors, and graph edges. They should work with operations such as remember, recall, build context, inspect decision history, or retrieve project state.

This agent layer is where things become more interesting, and also more difficult. In practice, the main value of an external memory system is not that a human can search old notes. It is that an agent can enter a task with some prior state, contribute new state while it works, and leave the system in a better condition than it found it. That sounds simple, but it forces a more disciplined interaction model.

A useful agent should interact with memory in three moments. At the start of work, it should retrieve context. During work, it should record only meaningful updates. At the end of work, it should write back a structured summary of what changed. These three moments sound almost trivial, yet most failures happen because one of them is skipped or handled badly.

At the start of a task, an agent should not simply run a broad semantic search and dump results into context. That usually produces noise. A better pattern is task-shaped hydration. The agent provides the task description, current workspace or repository, branch or module where relevant, and perhaps a few focus entities such as project names, files, or concepts. The memory system then returns a bounded context bundle: the latest summary, related decisions, recent unresolved issues, important conventions, and a small set of supporting source items. In other words, the agent should not fetch memory as raw data. It should request memory as working context.

During the task, the challenge changes. The risk is not lack of information but overproduction of it. Agents can generate observations constantly, but most of those observations do not deserve long-term persistence. If every intermediate guess, every transient failure, and every half-correct explanation is written into memory, the system degrades quickly. The answer is not to stop writing. It is to make writing more selective and more structured. The best candidates for persistent memory are design decisions, newly discovered conventions, recurring errors, architectural constraints, unresolved blockers, and concise summaries of meaningful work sessions. Temporary reasoning traces should usually remain temporary.

This is one of the core difficulties in agent memory. Models are verbose and opportunistic. They tend to treat anything they just generated as potentially valuable. A memory system has to resist that tendency. The interface should therefore encourage typed writes. Instead of a generic “save this,” the agent should write a decision, a blocker, a work-session summary, an observation, or a project-state update. That small amount of structure has a large effect later on. Retrieval becomes cleaner, provenance clearer, and summarization more reliable.

At the end of a task, the agent should write back something like a work-session record. This is more important than it may seem. If the system only stores raw notes and ad hoc observations, it never develops a coherent temporal thread. A structured session log can capture what the task was, what files or modules were touched, what decisions were made, what tests were run, what failed, and what remains open. This gives later agents something much closer to state than a bag of fragments.

Even with that pattern, there are real problems.

The first is obedience. Agents do not follow instructions consistently. One tool may recall memory at the start of work, another may ignore it, and a third may retrieve far too much. One may write back diligently while another forgets entirely. So memory behavior cannot rely only on polite prompting. It needs support from the environment. That can mean MCP tools with very clear semantics, wrapper commands that force hydration before a session starts, or end-of-task hooks that request and store a structured summary. The important point is that reliable memory behavior usually needs both instructions and outer control.

The second problem is ambiguity. Agents often do not know whether something should be stored, how important it is, or how it relates to what is already known. A model may discover a coding convention and store it as a universal rule when it was only true for one module. Or it may encounter an unresolved issue and describe it as a settled fact. This is where provenance and confidence matter. Memory entries should preserve source, time, author, and origin. Derived summaries should remain visibly derived. The system should not collapse source material, inference, and synthesis into one flat layer.

The third problem is fragmentation across agents. Different coding agents have different strengths, different tool surfaces, and different levels of discipline. One may be good at implementation detail, another at architectural discussion, another at exploration. If they all write memory in different shapes, the shared substrate becomes uneven. The way around that is to give them a small common vocabulary. The operations should be simple and stable: build context, remember, recall, log work session, retrieve project state. The memory system should normalize the rest.

The fourth problem is memory quality over time. Even if agents write structured records, the corpus still grows. Without periodic consolidation, the system accumulates repetition and drift. Summaries then become essential. Not summaries as a cosmetic feature, but as a maintenance mechanism. The system should periodically rebuild project summaries, topic summaries, and current-state views from source items, rather than relying forever on old synthetic text. This reduces decay and gives agents a much cleaner entry point.

The fifth problem is trust. Once agents begin using external memory, users may assume that retrieved context is current and correct. That assumption is dangerous. Memory should help agents reason, not replace reasoning. A retrieved item may be old, partial, or contradicted by a later decision. This is why memory results should carry status and provenance, and why the interface should distinguish between source records, linked relations, and synthesized current state. Good memory reduces uncertainty, but it does not abolish it.

These challenges are the main reason the memory layer should be treated as infrastructure rather than as a convenient plugin. It is not enough to store notes and let tools query them. Agents need a disciplined path into memory and a disciplined path back out. They need context bundles rather than noisy retrieval dumps. They need structured writes rather than free-form memory spam. They need summaries that can be rebuilt. And they need one canonical system that separates source, projection, and synthesis clearly enough to remain inspectable.

That is the line of thought behind Mindstate, which I have published as an experiment on my personal GitHub repository (see: [https://github.com/soyrochus/mindstate](https://github.com/soyrochus/mindstate) ). It is based on an earlier low-level implementation, but the current form matters more than the ancestry. The point is to test whether a practical external memory system can be built as a single application that combines three things at once: a low-level substrate shell, a higher-level memory API and user interface, and an internal MCP surface for external agents.

The low-level shell remains important. A memory system should not hide all of its internals behind a pleasant interface. It should still allow direct inspection of graph, vector, and relational layers for debugging, migration work, and brownfield reality. On top of that, Mindstate adds a higher-level application model based on canonical memory items with derived projections. Raw records remain primary. Embeddings, structural links, and summaries are downstream artifacts. That makes reindexing, re-extraction, and rebuilding possible when models or extraction logic change. It also makes provenance explicit rather than optional.

The third part is the MCP layer. This is what turns the application from a private tool into a shared substrate for coding agents and assistants. Different tools can read from the same memory and write structured work-state back into it. That does not solve everything. Agents are inconsistent, instructions drift, and automation can produce noise. This is one reason the project should be treated cautiously. It may fail in practice, or it may only work well under narrow conditions. But the underlying test is worth making: whether several tools can converge on one inspectable memory system instead of carrying isolated fragments of project state.

The implementation is deliberately conservative. PostgreSQL is the starting substrate, not because it is ideal for every layer, but because it is a strong and inspectable default. It handles canonical records, metadata, timestamps, and provenance well. With pgvector it can support semantic recall. With relational and graph-oriented structures it can support a useful structural layer. If later some part needs to split into a more specialized backend, that can happen. The important thing is that the system should depend on memory behaviors first, not on the accidental shape of a particular database.

This is the main correction to the Open Brain story. Externalized memory is the right direction, but openness alone is not enough. The difficult part is keeping one canonical corpus, deriving multiple views without drift, and exposing them through abstractions that remain usable for both humans and agents. Memory is not just stored context. It is typed, evolving, and partly synthetic. If the system does not make those distinctions clear, retrieval quality degrades and trust weakens.

So the position is simple. Nathan Jones is right about the problem and right about the need for external memory. The weakness is that public descriptions often stop too early, at the point where a prototype becomes visible. Mindstate is an attempt to push one step further and make the architecture more explicit: one application, one canonical model, several derived projections, a low-level shell, a higher-level API and UI, and an internal MCP interface for agents.

It is not presented as a finished answer. It is a fragile implementation hypothesis. But it tests a more serious version of the idea: that external AI memory should be built as infrastructure, not as a feature bolted onto a chat product.
